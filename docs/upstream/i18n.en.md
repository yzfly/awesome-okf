# Upstream proposal: i18n (`lang` / `canonical`)

- Status: **submitted**
- Issue: https://github.com/GoogleCloudPlatform/knowledge-catalog/issues/49
- Chinese draft: [okf-spec-zh.md 末尾的 i18n 提案](../okf-spec-zh.md#中文生态议题i18n-扩展提案草案)

## Issue text

> Title: Proposal: optional lang/canonical fields for multilingual concepts

### Summary

Add two optional frontmatter fields — `lang` and `canonical` — so a bundle can carry the same concept in more than one language. Purely additive: no MUST changes, no impact on existing bundles or consumers.

### Motivation

OKF v0.1 has no notion of language. Many bundles serve a mixed audience (an English source repo documented for a Chinese-speaking team, a glossary that needs both). There is no agreed way to say "this document is the Chinese version of that concept", so producers either drop a language or ship two documents that consumers see as unrelated. The spec already lets producers add keys (§4.1) and requires consumers to tolerate unknown ones (§4.1, §9); what's missing is a *shared name* for these two fields.

### Proposal

- `lang` — a BCP 47 language tag (`en`, `zh`, `zh-Hans`, …) for the body language. Absent = not declared.
- `canonical` — the concept ID of the primary-language version; translations point at a shared canonical concept.

Layout left to the producer — parallel files (`tables/orders.md` + `tables/orders.zh.md`) or parallel directories (`en/...` + `zh/...`).

Language-aware consumers SHOULD group documents sharing a `canonical` and select by `lang`; others treat each as independent (still conformant).

### Backward compatibility

No MUST changed; both optional; unknown to v0.1 consumers → ignored (§4.1, §9). Backward-compatible minor-version addition under §11.

### Reference implementation

https://github.com/yzfly/awesome-okf — producers emit `lang`; `canonical` links translations/originals.
