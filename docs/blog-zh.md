# 开放知识格式如何改善数据共享

> 原文:[How the Open Knowledge Format can improve data sharing](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/)
> 作者:Sam McVeety(数据分析技术负责人)、Amir Hormati(BigQuery 技术负责人),Google Cloud 数据云团队
> 发布日期:2026 年 6 月 12 日
> 译者:云中江树的AI伙伴

---

## 引子:开放知识格式(OKF)登场

基础模型一直在变强,但真正限制它们的,往往不是能力,而是缺少相关的上下文——尤其当人们开始用它们搭建智能体(agentic)系统的时候。这些模型能帮你写代码、总结文档、分析数据集,可它们仍然需要拿到对的信息,才能给出准确、可落地的结果。

正因如此,今天我们推出**开放知识格式(Open Knowledge Format,OKF)**:一份开放规范,把"LLM 维基(LLM-wiki)"这一模式正式确立为一种可移植、可互操作的格式。它中立于厂商,既对智能体友好,也对人友好,用来表达现代 AI 系统所需要的元数据、上下文与经过梳理的知识。

按照目前发布的形态,**OKF v0.1** 把知识表示为一个目录,里面是一份份带 YAML 头信息(frontmatter)的 Markdown 文件,再加上一小套约定俗成的规则——正是这套规则,让不同生产者写出的维基,能够被不同的智能体直接消费,无需任何翻译转换。

就这么简单。没有复杂的压缩方案,没有新的运行时,也不强制任何 SDK。一束 OKF 文档,本质上是:

- **不过是 Markdown**——任何编辑器都能读,GitHub 上能直接渲染,任何搜索工具都能建索引;
- **不过是文件**——可以打包成 tarball 分发,可以托管在任意 git 仓库,可以挂载到任意文件系统;
- **不过是 YAML 头信息**——只为那一小撮需要被查询的结构化字段服务:类型(type)、标题(title)、描述(description)、资源(resource)、标签(tags)和时间戳(timestamp)。

如果你用过 Obsidian、Notion、Hugo,或是过去一年里冒出来的各种 LLM 维基玩法,这副样子你会觉得眼熟。OKF 所做的,只是把让这些做法彼此互通所必需的那一小套约定,正式定下来。

下面我们就来看看:OKF 能为你的组织解决什么问题,它如何运作,如何上手,以及接下来会怎样。

## 一片支离破碎的上下文版图

在大多数组织里,基础模型要用到的信息,绝大部分是**内部知识**:一张表的 schema、某个指标在你们业务里的真正含义、某次故障的处理手册、两套系统之间的关联(join)路径、一个旧 API 的弃用通知,等等。

而今天,这些知识的"原子",散落在五花八门、彼此割裂的系统里:

- 各有各 API 的元数据目录(catalog);
- 维基、第三方系统,或者共享盘;
- 代码注释、文档字符串(docstring),或者 notebook 单元格;
- 还有,几位资深工程师的脑子里。

当一个 AI 智能体需要回答"如何从我们的事件流里算出周活跃用户?"时,它不得不从这些零散、互不兼容的入口里,把答案一点点拼凑出来。每家厂商都提供自己的目录、自己的 SDK、自己的知识图谱 schema,而没有一份知识能在产品之间、组织之间轻松搬动。

结果就是:每一个智能体开发者,都在从零开始解决同一个"上下文拼装"问题;每一家目录厂商,都在重新发明同一套数据模型;而知识本身,被锁死在最初创造它的那个界面背后。

## 把知识当作一座活的维基

开发团队正在改变搭建 AI 智能体的方式。与其让模型一遍又一遍地在相同文档里翻找相同事实,不如给智能体一座共享的 Markdown 知识库——它会随时间越用越好用。这样一来,智能体就能接手"读文件、改文件"这类苦差事,而你的团队则负责梳理内容,像管理代码一样管理它。

知名 AI 研究者与教育者 Andrej Karpathy,在他那篇 LLM Wiki 短文里,把这个想法说得最为干脆。他写道:"LLM 不会觉得无聊,不会忘记更新某处交叉引用,还能一次性改动 15 个文件。"——那些让人类放弃个人维基的繁琐记账活儿,恰恰是 LLM 的拿手好戏。

类似的"知识即维基"模式,正以不同的名字反复出现:接上编码智能体的 Obsidian 仓库(vault)、`AGENTS.md` / `CLAUDE.md` 这一族约定文件、塞满了 `index.md` 和 `log.md` 的仓库(智能体动手干活前会先去查阅),以及数据团队内部的"元数据即代码(metadata as code)"仓库。

这个模式既诱人又强大,但每一个实例都是定制的。Karpathy 的维基、你们团队的维基、某家厂商导出的目录,看上去可能都一个样(Markdown、头信息、交叉链接),但它们当初谁也没打算跟别人配合。对于"每份文档该带哪些字段""哪个文件名代表什么含义",并不存在一个公认的答案。于是,维基里编码的知识,始终被困在最初那支团队内部;每当有人要搭一个新智能体,就得把同样的活儿重做一遍。

## 缺的是一种格式,而不是又一个服务

这个问题的答案,不是再来一个知识服务。你需要的是一种**格式**——一种这样来表示知识的方式:

- 人人都能生产,无需 SDK;
- 人人都能消费,无需做集成;
- 在系统、组织、工具之间搬来搬去也不会失真;
- 能和它所描述的代码一起,放进版本控制里;
- 对人可读,对智能体可解析:同一份文件,不需要任何翻译层。

OKF 在设计上,就是这样一种格式。

## OKF 如何运作:一屏看懂的设计

一个 OKF **包(bundle)**,就是一个装着 Markdown 文件的目录,每个文件表示一个**概念(concept)**:任何你想记下来的东西都行,包括表、数据集、指标、操作手册(playbook)、应急手册(runbook)、API。一个概念对应一个文件。文件路径,就是这个概念的身份标识:

```
sales/
├── index.md
├── datasets/
│   ├── index.md
│   └── orders_db.md
├── tables/
│   ├── index.md
│   ├── orders.md
│   └── customers.md
└── metrics/
    ├── index.md
    └── weekly_active_users.md
```

每份概念文档,有一小块 YAML 头信息承载结构化字段,其余一切则交给 Markdown 正文:

```
---
type: BigQuery Table
title: Orders
description: 每行记录一笔已完成的客户订单。
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, revenue]
timestamp: 2026-05-28T14:30:00Z
---

# Schema

| 列名 | 类型 | 说明 |
|---|---|---|
| `order_id` | STRING | 全局唯一的订单标识符。 |
| `customer_id` | STRING | 外键,指向 [customers](/tables/customers.md)。 |

# Joins(关联)

通过 `customer_id` 与 [customers](/tables/customers.md) 关联。
```

概念之间用普通的 Markdown 链接互相连接,于是整个目录就变成了一张**关系图谱**——它比文件系统所隐含的那种父子层级链接要丰富得多。包里还可以选择性地放入 `index.md` 文件(供智能体在层级中导航时**渐进式展开**信息),以及 `log.md` 文件(记录变更的时间线历史)。

完整的 v0.1 规范(包括符合性标准、交叉链接规则,以及为数不多的几个保留文件名)——一页纸就装得下。

## 设计背后的三条原则

**1. 尽量不替你做主。** OKF 对每个概念只有一个硬性要求:必须有一个 `type` 字段。其余的一切——存在哪些类型、还要带哪些字段、正文分哪些章节——全部交给生产者自己决定。规范定义的是"互操作的接触面",而不是"内容模型"。

**2. 生产者与消费者各自独立。** OKF 把"谁写知识"和"谁用知识"干净利落地分开。人手写的包,可以被 AI 智能体消费;元数据导出流水线生成的包,可以放进可视化工具里浏览;一个 LLM 合成的包,可以被另一个 LLM 查询。格式是契约;两端的工具,各自都能随意替换。

**3. 是格式,不是平台。** OKF 不绑定任何特定的云、数据库、模型厂商或智能体框架。它永远不会要求你为了读、写、提供它而注册某个专有账号或装某个 SDK。我们之所以把它作为开放标准发布,是因为一种知识格式的价值,来自有多少方在说这门语言,而不是来自它归谁所有。

## 我们随规范一起交付了什么

为了让这个格式变得具体,我们在生产端和消费端都发布了**参考实现**:

- 一个**富化智能体(enrichment agent)**:它会遍历一个 BigQuery 数据集,为每张表和每个视图起草一份 OKF 概念文档,然后再跑一遍 LLM,去爬取权威文档,为每个概念补上引用、schema 和关联路径。
- 一个**静态 HTML 可视化器**:它能把任意 OKF 包,变成一个单一、自包含文件里的交互式图谱视图;查看端无需后端、无需安装,任何数据都不会离开这个页面。
- **三个开箱即浏览的示例包**:GA4 电商、Stack Overflow、Bitcoin 公共数据集——由参考智能体生成,并提交进仓库,作为符合 OKF 规范的、活生生的范例。

这些都是刻意做成"概念验证"的。富化智能体只是演示了生产 OKF 的一种方式——格式本身并不要求特定的智能体框架或 LLM;可视化器只是演示了消费它的一种方式——格式本身并不要求 HTML 或图谱视图。我们预期(也希望!)生产者与消费者的生态,会远远长得超出我们已经交付的这些。

## 接下来去往何方

OKF v0.1 是一个起点,不是一份完成了的标准。随着越来越多的生产者和消费者出现,随着我们共同摸清智能体在实践中究竟需要怎样的知识表示,这个格式会不断演化。

我们从第一天起就在公开场域发布它,因为只有这样,一种知识格式才配得上"知识格式"这个名字——无论你是在做一个知识目录、一条富化流水线、一座为 AI 智能体量身定做的维基,还是 AI 知识领域里的任何东西。

从这里起步,我们鼓励你:

- **读规范**(很短的!);
- **写一个生产者**:为你的源系统、你的数据库、你的文档站点;
- **写一个消费者**:一个查看器、一个搜索索引,或一个能在包之上做推理的智能体;
- **拿参考实现**对着你自己的数据**试一试**;
- **提 issue、发 PR,或者提出扩展提案**:规范是带版本号的,并且明确为"向后兼容地生长"而设计。

仓库、规范和示例包都已放在 GitHub 上。我们也已经升级了 Google Cloud 的知识目录(Knowledge Catalog),让它能够摄入开放知识格式,并把它提供给我们的智能体。相关代码和示例都可以找到。

格式本身,才是真正的贡献。我们交付的那些工具,只是为了让它落地成真,为了把"试一试"的成本降到最低。无论你今天的知识长成什么样子,OKF 的设计目标,就是成为那门"通用语(lingua franca)"——明天,你的知识可以兑换成它。
